1. Introducción

1.1. Propósito

TConversacion es un componente Svelte de alto nivel, disponible en el paquete @jeff-aporta/svelteprueba, diseñado para encapsular y abstraer la totalidad de la lógica de una interfaz de conversación en tiempo real.

Su objetivo principal es proporcionar una solución "plug-and-play" que minimice drásticamente el tiempo de desarrollo. Al consumir este componente, el desarrollador se abstrae de la gestión manual de:

  • Conexiones WebSocket y reconexiones (vía Azure Web PubSub).
  • Gestión de estado de mensajes (enviando, enviado, editado, leído).
  • Paginación y carga de historial (scroll infinito).
  • Renderizado condicional de tipos de mensajes (citas, reacciones, eliminados).
  • Lógica de UI para citar, editar y reaccionar a mensajes.

1.2. Arquitectura de Consumo

Para integrar TConversacion, es fundamental comprender la interacción de las tres entidades del ecosistema:

  1. ISW-Chat (Frontend - El Consumidor):
    • Es el proyecto (Astro/Svelte) que importa y renderiza TConversacion.
    • Es responsable de instanciar y configurar el TMensajeClient con las credenciales y la URL del backend.
  2. ISS (Backend API - El Servidor):
    • Una API de Azure Functions que expone la lógica de negocio.
    • API de Persistencia (REST): Provee endpoints CRUD (Crear, Leer, Actualizar, Borrar) para la entidad Mensaje.
    • Broker de Conexión (WebSocket): Expone un endpoint (ej. /api/pubsub) que genera una URL de conexión temporal a Azure Web PubSub. TConversacion utiliza esta URL para establecer la conexión en tiempo real.
  3. ISP-Chat (Lógica/Tipos - El Paquete):
    • Un paquete NPM (@jeff-aporta/ispchat) que actúa como librería compartida para mantener la consistencia contractual entre frontend y backend.
    • Contratos de Datos (Tipos): Define las interfaces TypeScript (ej. TMensaje, TConversacion, TReaccion) que ambos, ISW e ISS, utilizan.
    • Cliente API: Exporta la clase TMensajeClient, un *wrapper* que encapsula la lógica fetch para interactuar con la API de ISS.

1.3. El Patrón de Diseño: Inversión de Dependencias

El diseño de TConversacion se basa fundamentalmente en el principio de Inversión de Dependencias (la "D" de SOLID) para lograr un Acoplamiento Débil.

La Solución (Acoplamiento Débil): La arquitectura invierte esta dependencia. En lugar de que la Vista dependa de la Lógica, ambos dependen de una abstracción (la Interfaz).

  1. La Vista (TConversacion): Solo conoce el "Contrato" (la interfaz). Su única responsabilidad es renderizar la UI y llamar a los métodos de ese contrato (ej. props.controlador.fnEnviarMensaje()).
  2. El Contrato (TMensajeClientInterface): Define el "QUÉ" (qué métodos deben existir), pero no el "CÓMO" (cómo se implementan).
  3. El Controlador (Su implementación): Es la lógica concreta que el consumidor crea e inyecta en el componente. Aquí es donde vive el fetch, la lógica de autenticación y el manejo de URLs.

Este patrón (conocido como Inyección de Dependencias) es la base de esta arquitectura, permitiendo que el componente sea testeable, mantenible y agnóstico a la fuente de datos.

2. Paso 1: El Contrato del Cliente (TMensajeClientInterface)

Como se describe en el patrón de Inversión de Dependencias, al componente TConversacion no le debe importar cómo se implementa la lógica de backend (es decir, cómo se hacen las llamadas API, cómo se gestiona la autenticación, o qué librería se usa).

Al componente solo le debe interesar que el objeto mensajeClient que se le inyecta (como prop) cumpla con un "Contrato" formal. Este contrato es la interfaz TMensajeClientInterface, la cual se alinea con una metodología de "Diseño Basado en Interfaz" (Interface-First Design).

Consideración Crítica: Definición de Altura

Observe el <div> que envuelve a TConversacion con style="height: 500px;". Esto es fundamental. El componente de chat está diseñado para llenar el 100% de la altura de su contenedor padre (height: 100% por defecto). El área de historial de mensajes necesita una altura definida para saber cuándo debe volverse "scrollable".

Si no defines una altura en el contenedor padre, el chat intentará crecer verticalmente de forma infinita a medida que se cargan mensajes, en lugar de crear una barra de scroll interna.

4. Paso 3: API de Propiedades (Props)

A continuación, se detallan todas las propiedades (props) expuestas al consumidor, analizadas desde Conversacion.svelte y Historial.svelte.

4.1. Props Requeridas (Core)

Estas propiedades son mandatorias para la inicialización y funcionalidad del componente.

Prop Tipo Descripción
mensajeClient TMensajeClientInterface (Mandatoria) La instancia configurada del cliente. Es el motor para toda la persistencia y la obtención de la URL de PubSub.
iusuarioActual string (Mandatoria) El identificador único del usuario actual. Se utiliza internamente para:
1. Determinar la alineación de los mensajes (enviados vs. recibidos).
2. Registrar lecturas (no se auto-registra la lectura).
3. Habilitar permisos de edición y eliminación (solo en mensajes propios).
4. Identificar la autoría de reacciones.
idConversacion string (Mandatoria) El identificador único de la sala de chat. Se utiliza internamente para:
1. Solicitar el historial de mensajes correcto a la API.
2. Unirse al grupo de Web PubSub correspondiente para recibir eventos en tiempo real.
ttdconversacion TTDTipoConversacion (Mandatoria) Un enum que define el modo de renderizado.
Valor: TTDTipoConversacion.chat.
Actúa como un selector de modo. Aunque actualmente solo chat está implementado, esta prop permite la extensibilidad futura (ej. TTDTipoConversacion.foro).

4.2. Props Opcionales (Control de UX)

Estas propiedades ajustan el comportamiento y la experiencia de usuario. Se definen en TConversacion y se propagan internamente a TChat y THistorial.

Prop Tipo Default Descripción
enviarConEnter boolean true Define el comportamiento de la tecla "Enter" en el campo de entrada.
true: "Enter" envía el mensaje. "Shift+Enter" crea una nueva línea. (Comportamiento de WhatsApp/Telegram).
false: "Enter" crea una nueva línea. "Ctrl+Enter" (o "Cmd+Enter") envía el mensaje. (Comportamiento de Slack/Discord).
mostrarAvatarUsuario boolean true Controla la visibilidad del componente TAvatar junto a los mensajes.
true: Renderiza el TAvatar.
false: Oculta el TAvatar para una interfaz más minimalista.
avatarAncho string | number 18 Define el ancho (y alto) del TAvatar. Acepta un número (ej. 32) que se interpreta como px, o un string (ej. "2.5rem").
mostrarNombreSobrePrimerMensaje boolean true Controla la agrupación visual de mensajes.
true: Agrupa mensajes consecutivos del mismo usuario, mostrando el nombre del autor solo sobre el primer mensaje del bloque.
false: Muestra el nombre del autor en cada mensaje.
mostrarCabecera boolean true Controla la visibilidad de la TTarjeta superior que envuelve el slot="cabecera".
true: Renderiza el espacio de la cabecera.
false: Oculta la cabecera, y el historial de mensajes ocupa el espacio superior (layout de 2 filas en lugar de 3).

4.3. Props Opcionales (Control de Estilo)

Estas propiedades permiten la personalización visual del componente.

Prop Tipo Default Descripción
style string | Record<...> "" Aplica estilos CSS en línea al componente raíz TTarjeta (el contenedor principal). Es el método recomendado para definir la altura (ej. style={{ height: "80vh" }}).
styleHistorial string | Record<...> "" Aplica estilos CSS en línea directamente al contenedor del historial (THistorial). Es ideal para backgroundImage (ej. styleHistorial={{ backgroundImage: 'url(...)' }}).
historialThemeName string | undefined undefined Asigna el atributo data-theme-name al componente THistorial. Útil para aplicar temas CSS personalizados (ej. "dark", "light") si están definidos en el proyecto consumidor.

Caso 2: Ejemplo Avanzado (Reemplazo Total del Formulario)

Advertencia: Si se inyecta contenido que no es un simple input (ej. botones propios, divs envoltorios), TConversacion asume que el consumidor tomará control total del formulario.

Esto significa que el consumidor es ahora responsable de:

  1. Renderizar sus propios botones (Adjuntar, Enviar).
  2. Capturar el evento de envío (clic en botón o "Enter").
  3. Instanciar el objeto TMensaje (o TCita).
  4. Llamar a mensajeClient.fnEnviarMensaje(nuevoMensaje) manualmente.
  5. Limpiar el input y el estado de la cita/edición.

7. Paso 6: Solución de Problemas (Troubleshooting)

7.1. Error: "Cannot read properties of undefined (reading 'fnEnviarMensaje')"

  • Causa: La prop mensajeClient es undefined, null, o no es un objeto que implemente la interfaz TMensajeClientInterface.
  • Solución: Asegurarse de que el objeto pasado a mensajeClient esté correctamente instanciado y contenga todas las funciones requeridas por la interfaz antes de renderizar TConversacion.

7.2. Error de CORS (Cross-Origin Resource Sharing)

  • Causa: El servidor ISS (Azure Functions) no ha sido configurado para aceptar peticiones desde el origen del frontend (ej. http://localhost:4321 o https://mi-app.com).
  • Solución: En la configuración de la Azure Function App (servidor ISS), navegar a la sección "CORS" y añadir el origen del frontend (ej. http://localhost:4321) a la lista de orígenes permitidos.

7.3. El chat se conecta, pero los mensajes en tiempo real de otros usuarios no llegan

  • Causa: La conexión WebSocket está activa, pero el cliente no se ha unido al grupo (sala) correcto. Esto casi siempre es debido a un idConversacion incorrecto o diferente entre los usuarios.
  • Solución: Asegurarse de que ambos usuarios (en diferentes navegadores) estén usando exactamente el mismo idConversacion (sensible a mayúsculas/minúsculas).

8. Paso 7: Checklist de Implementación

  • [ ] Importar TConversacion, TTDTipoConversacion y la interfaz TMensajeClientInterface.
  • [ ] Crear o inyectar una instancia de un cliente que implemente TMensajeClientInterface.
  • [ ] Definir iusuarioActual e idConversacion. El consumidor define cómo se establecen estos valores.
  • [ ] Implementar el componente TConversacion en el HTML pasando todas las props requeridas: mensajeClient, iusuarioActual, idConversacion, y ttdconversacion.
  • [ ] (Opcional) Personalizar con styleHistorial o slot="cabecera" para alinear con la identidad de tu app.
